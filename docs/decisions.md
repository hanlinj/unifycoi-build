## Phase 13 — Postgres migration

### ADR-013-01 · Data layer migrates SQLite→Postgres and raw SQL→Kysely, staged module-by-module, before real customer data exists (2026-07-10)
- **Decision:** Migrate the app's data layer off `better-sqlite3` onto Postgres, adopting Kysely (typed query builder) in the same pass rather than staying raw-SQL or introducing a heavier ORM. Target host: Railway (managed Postgres). This is a plan-first, staged migration — module by module, never a big-bang rewrite — with the SQLite baseline test suite as the correctness contract at every checkpoint (1071 tests as of the Stage 4 commit, not the 1010 figure earlier reports cited — see `docs/phase-13-postgres-migration.md`'s baseline-correction note). Each stage is independently committable and must leave the app fully working.
- **Why now, why in one pass with the query-builder switch:** the database is empty (no live customer data), which is the cheapest and safest point in the project's life to absorb a storage-layer change — this same reasoning already justified a fresh-baseline-over-migration-porting choice elsewhere in this phase (see the investigation report for the migrations question). Bundling the Kysely adoption into the same pass avoids doing the sync→async rewrite twice (once bare, once again when a query builder is added later) — the async ripple is the dominant cost either way, so it should be paid once, deliberately, with type safety as a direct benefit rather than a follow-up nice-to-have.
- **Context:** Phase 13 kickoff, investigation-only pass (no code changed). A full inventory of ~101 files / ~430 direct DB calls across 13 logical modules, a catalog of SQLite-specific constructs (COLLATE NOCASE email lookups, INSERT OR IGNORE/REPLACE, four `rowid`-ordering call sites with no Postgres equivalent, TEXT-as-boolean and TEXT-as-timestamp columns with no CHECK constraints), and a sync→async blast-radius estimate (roughly 250-350 await insertions across ~110-120 files, plus ~13 genuinely structural rewrites at transaction boundaries and N+1-in-map sites) were produced to inform the staged plan. See the investigation report (delivered in chat, not duplicated here) for the full module map, the concrete rework list, and the recommended stage order.
- **Scope boundary:** foundation only — when complete, the app must look and behave identically; this is not bundled with any product/design change. `BlobStore` (Backblaze B2 + filesystem adapter) and the background workers' *orchestration* are confirmed independent of the SQL engine choice and are out of scope, though the workers' own DB calls (mostly already-async functions with synchronous `better-sqlite3` calls inside) do get touched as part of whichever module owns their tables.
- **Known hard spots flagged up front, not deferred:** (1) the `billing_snapshots`/`audit_events` `rowid`-tiebreaker pattern needs a real identity/serial column before cutover, in four call sites; (2) `TenantDB.transaction()` and its 9 call sites rely on better-sqlite3's synchronous-callback transaction API (already documented in-repo as the reason the Stripe quantity-sync worker runs as a separate poller) — Kysely/Postgres transactions are inherently async, so every transaction boundary is a structural rewrite, not a mechanical `await` add; (3) `COLLATE NOCASE` email/name lookups (8 call sites) need a `lower()`-based rewrite or `citext`; (4) the test suite's true-in-memory-SQLite-per-test isolation (`tests/helpers.ts:setupTestDb`) has no free equivalent on Postgres and needs a deliberate per-test isolation mechanism (transactional rollback or a uniquely-named ephemeral database per run).
- **Confirmed (Stage 0, 2026-07-10): fresh baseline, not a ported one-for-one migration; Kysely, not raw `pg`.** Both were investigation-phase recommendations; Stage 0 built and proved them, so they're adopted, not just proposed. The baseline (`src/migrations-pg/001_baseline.ts` — moved here from Stage 0's temporary `db-postgres/` scaffolding once Stage 1 wired it for real) is the cumulative end-state of the 17 SQLite migrations expressed in idiomatic Postgres (booleans as `boolean`, timestamps as `timestamptz`, JSON payloads as `jsonb`, and a real `seq bigserial` tiebreaker column on `billing_snapshots`/`audit_events` replacing `rowid`) — verified column-by-column against all 17 files directly, not from a summary. One concrete finding Stage 0 surfaced that refines point (2) above: Kysely's simple callback-style `.transaction()` does NOT auto-nest via savepoints (it throws if called again on an already-open `Transaction`) — only the CONTROLLED transaction API (`db.startTransaction()` + `.savepoint(name)`) supports nesting, so `TenantDB.transaction()` in Stage 1 must be built on that API and must itself detect "already inside a transaction" to decide whether to open a new transaction or take a savepoint. Test isolation is solved as a two-layer scheme: one ephemeral database per test run (cloned via `CREATE DATABASE ... TEMPLATE` from a schema-only template, isolating concurrent runs/worktrees from each other) plus a rolled-back transaction per test within that run (isolating tests from each other and reconstructing `:memory:`'s throwaway-per-test guarantee) — proven under concurrent execution (two isolation-proof runs launched simultaneously, one from the main tree and one from a separate `git worktree`, against the same Postgres server, each got its own database and neither saw the other's data). Local Postgres 16 (apt-installed on this host) stands in for Railway during Stage 0/1 development; Railway itself is deferred to a later stage per the owner's call, not blocking foundation work.
- **Stage 6a · Vendor invite tokens: revoke-on-issue (deliberate security change, NOT behavior-preservation — flagged explicitly, not buried in a conversion diff).** Before this stage, a vendor invite token (`invites` table — a completely separate mechanism from `password_reset_tokens`, used only by the tokenized vendor portal) was reusable for its full 14-day TTL with no consumption/invalidation step at all: `resendInvite` minted a fresh token without revoking the old one, so two (or more, after repeated resends) tokens could be simultaneously live for the same vendor. As of Stage 6a, issuing ANY new invite for a vendor — via `createVendorInvite` (new vendor), `resendInvite` (resend), or `applyDecision`'s `request_correction` branch — first revokes every other still-live prior invite for that vendor, inside one transaction, before inserting the new row. Revocation is vendor-wide across ALL purposes (an onboarding invite request-correction'd away is revoked too, not just prior corrections) — the strongest, least-surprising posture: only the newest token for a vendor is ever valid, full stop. TTL itself is unchanged (still 14 days for both onboarding and correction).
  - **Mechanism:** a new nullable `revoked_at timestamptz` column on `invites` (added by amending the Stage 0/1 baseline directly — the DB has no real data yet, so no migration-with-backfill was needed); a single new choke point, `issueInviteToken()` (`src/lib/auth/invite-token.ts`), that every mint site now routes through — no mint site writes `invites` directly anymore. The revoke query (`UPDATE invites SET revoked_at = now() WHERE tenant_id = $1 AND vendor_id = $2 AND revoked_at IS NULL AND token_expires_at > now()`) needs `IS NULL`/`>` predicates `TenantDB.update()` can't express (it only builds `col = val` equality), so it drops to the Kysely builder directly inside `withTransaction()`, manually re-applying the same tenant_id-scoping discipline `TenantDB`'s own helpers enforce internally.
  - **Split across the seam, deliberately:** Stage 6a (this stage, operator-side: `vendors.ts`, `resend-invite.ts`, `decision.ts`) only ever WRITES `revoked_at`. Reading it — `validateInviteToken()` (`src/lib/services/vendor-token.ts`) rejecting a token whose `revoked_at IS NOT NULL` — is explicitly Stage 6b's job (the portal side), not touched here. Until 6b lands, a revoked token's OLD behavior (still validates successfully) is unchanged; the column exists and gets written correctly, but nothing reads it yet. This is a real, temporary gap, not an oversight — flagged so Stage 6b's kickoff doesn't have to rediscover it.
  - **Verified, not assumed:** dedicated tests prove the invariant across all three mint paths — the prior invite's `revoked_at` gets set (not deleted — the row survives for audit purposes), exactly one live (`revoked_at IS NULL`) invite remains per vendor after any issuance, revocation reaches across purposes (an onboarding invite is revoked by a correction request), and revocation is strictly vendor-scoped (issuing for vendor B never touches vendor A's live invite) — `tests/vendors-operator.pg.test.ts`.

- **Stage 8b · Notifications converted (`worker.ts`, `digest.ts`, `resend-webhook.ts`, `renewal.ts`'s remaining `applyExpirationFlip`, `manual-reminder.ts`) — mechanical, no architecture change, plus one checked-and-declined correctness question and one documentation-integrity fix.** Confirmed independent of 8a (retention) and 8c (audit exports) per the pre-Stage-8 scoping trace — no shared claim helper, no forward dependency either direction.
  - **The claim was checked for a concurrency gap and found not to need one — auditable, not asserted.** `processDueNotifications`'s claim is a non-binding discovery `SELECT` followed by a per-row guarded `UPDATE notifications SET status='sending' WHERE id=$1 AND status='queued'`. Correctness rests entirely on that UPDATE: Postgres evaluates the `WHERE` predicate transactionally at the row level, so under any concurrency — overlapping ticks in one process, or genuinely separate app instances — at most one caller's UPDATE can see `status='queued'` and flip it; every other concurrent claimer gets `numUpdatedRows === 0` and skips. This was already true of the pre-conversion SQLite version and survives the mechanical Kysely translation unchanged (`.updateTable('notifications').set(...).where('id','=',id).where('status','=','queued').executeTakeFirst()`, `Number(numUpdatedRows)`). **`FOR UPDATE SKIP LOCKED` was deliberately NOT introduced** — that pattern earns its keep collapsing discovery+claim into one multi-row statement to avoid wasted contention under heavy multi-instance load, a throughput optimization for a shape this code doesn't have, not a fix for a correctness gap that doesn't exist here. Corroborated independently by `docs/launch-prep.md`'s **OPS-6**, written in Phase 7 before this migration existed, which already drew the identical line: "the notification + export workers are claim-guarded, but digest/retention are not leader-elected." **Verified, not assumed:** a dedicated test opens two genuinely independent Postgres connections (`test-isolation.ts`'s `createEphemeralTestDatabase`/`kyselyFor`, not the shared per-test transaction `setupTestDb()` uses — a row uncommitted on one connection is invisible to a second, so the standard harness can't host a real cross-connection race) and fires the claim UPDATE concurrently on both against the same row: exactly one gets `numUpdatedRows === 1`, the other `0`. See `tests/notifications-worker.pg.test.ts`.

### ADR-012-10 · Tenant cockpit rate/fee edits: Stripe-first ordering, immutable-Price swap, subscription-item partial-update reliance (2026-07-09)
- **Decision:** `/platform/tenants/[id]` (Slice 6) gets four controls beyond view: resend the first Admin's credential invite, resend the billing-setup link, edit the per-location rate, edit the one-time setup fee. No new schema — `applied_template_id` (Phase 3) was already a column, just never selected by `getTenantById`; no rate-history table was added (the audit trail already covers it, see below).
- **Rate edit ordering — Stripe first, local DB second, always:** `updateTenantRate` calls `BillingProvider.updateSubscriptionPrice` BEFORE touching `tenants.monthly_rate_cents`. If Stripe rejects the swap, the local column is not written at all — the function returns a structured `{ pushedToStripe: false, error }` (same non-throwing shape as `attachBilling`), never leaves a local rate Stripe isn't actually charging. Verified directly: after an injected Stripe failure, the DB read back is byte-identical to the pre-edit value, and no `tenant.settings_changed` audit event fires.
- **Why a new Price every time, never an edit:** Stripe Prices are immutable (`unit_amount` cannot be changed on an existing Price) — same reason `createSubscription` already creates a fresh Price at provisioning. A rate edit is always create-Price + repoint-the-subscription-item, `proration_behavior: 'none'` (identical rule to `updateSubscriptionQuantity`) so the new rate takes effect at the NEXT billing cycle, never a mid-month partial charge.
- **Quantity-sync interaction — the highest-risk check in this slice:** the price-swap update (`items: [{id: itemId, price: newPriceId}]`) deliberately omits `quantity`. Stripe subscription-item updates are partial (only passed fields change), so the item's current quantity — owned by `quantity-sync.ts`'s poller — survives untouched, and the item's own `id` never changes across a price swap (this UPDATEs the existing item, it doesn't delete+recreate it), so `updateSubscriptionQuantity`'s own `items.data[0]?.id` lookup keeps resolving the same item afterward. Reasoning rests on Stripe's documented partial-update semantics and the fact that `updateSubscriptionQuantity` already relies on the identical assumption in reverse (omitting `price` to leave it alone) — **not yet re-verified against a live Stripe test-mode call this session** (no key available); carried on `docs/launch-prep.md`'s ledger to fold into the next manual Stripe click-through.
- **Setup-fee guard gates on `stripe_subscription_id IS NOT NULL`, not `lifecycle_state`:** the fee is committed to Stripe as a pending invoice item synchronously inside `attachBilling`, the moment a subscription is created — before the tenant necessarily pays (activation waits on the separate `invoice.paid` webhook). A tenant can have a live subscription while still `lifecycle_state='provisioning'`, and the fee is already un-editable at that point; gating on lifecycle state alone would have let an edit through that Stripe would silently ignore. Blocked with an explicit `blockedReason` surfaced to the operator, not a silent no-op.
- **Live subscription status, not a local derivation:** added `BillingProvider.getSubscriptionStatus` and call it on every cockpit page load. Considered "is one more live Stripe call per load too heavy" and concluded no — this is a low-traffic, platform-operator-only screen already making one live call (`listRecentInvoices`); an operator screen whose job is showing real payment state must not read a stale local "active" when a customer's card has actually failed.
- **`billing.rate_synced_to_stripe`** logged as its own audit event (distinct from `tenant.settings_changed`, which `updateTenant` already logs with a from/to payload) — same reasoning `attachBilling` logs `billing.subscription_created` separately from `billing.customer_attached`: a future audit can tell "did this reach Stripe" apart from "did the DB column change."
- **Resend controls reuse token primitives directly, not `sendUserInvite`:** `resendFirstAdminInvite`/`resendBillingSetupLink` call `issueInviteToken`/`issueBillingSetupToken` directly rather than 5b's `sendUserInvite`, because that function's audit call hardcodes `actorType:'user'` — reusing it verbatim would misattribute a platform-initiated resend as a tenant-user action.
- **Context:** Slice 6. Closes the OPS-14 remainder in full (both the invite half from 5b and the billing-setup/rate half here).

### ADR-012-09 · `users.invite_sent_at` — a new column, same treatment as `purpose` (2026-07-09)
- **Decision:** Added `invite_sent_at TEXT` (nullable) to `users` (migration 017) so the Users panel can distinguish "invited, no link ever sent" from "invited, a link has already gone out at least once" — the Send-invite vs Resend-invite button label. `status` alone can't carry this: a user stays `'invited'` from creation all the way until they actually set a password, regardless of how many invite links were issued to them in between.
- **Backfill, deliberately:** every pre-existing row gets `NULL` (SQLite's implicit default for a column added with no `DEFAULT` clause) — correctly meaning "never sent," since nothing retroactively sent anyone a link by this migration running. The only writer is `sendUserInvite` (`src/lib/services/users.ts`), which sets it explicitly to the current timestamp on every call — send and resend are literally the same function call; there is no separate "first send" code path to fail to set it.
- **Why logged as its own entry, not folded into the slice summary:** same standard applied to ADR-012-06 (`password_reset_tokens.purpose`) — a schema change gets its own record before the slice closes, not a passing mention. Named explicitly here so a future reader hitting this column doesn't have to reverse-engineer its meaning from the migration comment alone.
- **Resend does NOT invalidate the previous outstanding token.** `sendUserInvite` calls `issueInviteToken` (Slice 4's function, unmodified) on every send, which never revokes sibling tokens for that user — same shape the Admin invite and billing-setup links already accept. This means a dormant user can have more than one *live* invite token at once after a resend. That's an accepted, pre-existing pattern, not new risk introduced here: `confirmPasswordReset` already invalidates every other outstanding reset/invite token for a user the moment ANY one of them is actually used, so there is never more than one *usable* outcome — only, transiently, more than one link that would produce it. Verified directly: after a resend, using the FIRST token still succeeds, and the second (still-outstanding) token is invalidated as a side effect (`tests/send-user-invite.test.ts`).
- **Verified, not assumed (the two things this slice's brief asked to check):** (1) `sendUserInvite` reuses `issueInviteToken` verbatim — the exact same function the provisioning wizard's Admin invite and `activateTenantOnFirstPayment` call; no second issuer was written. (2) confirmed end-to-end, not just by inspection, that the invited→active flip holds for an **import-created** manager specifically (not only admin-provisioned users): `bulkCreateLocationsWithManagers` → `sendUserInvite` → `peekResetToken` (lands `valid`+`invited`) → `confirmPasswordReset` (flips to `active`) → `loginResolvingTenant` succeeds, all in one test (`tests/checkpoint-bulk-import-to-active-login.test.ts`).
- **Context:** Slice 12/5b, Feature 2. Closes the Slice-4a-era ledger item ("invite retrieve/resend, no home yet") and the invite half of launch-prep's OPS-14 — the panel is `/users`, wiring `sendUserInvite` behind `POST /api/users/:id/send-invite`.

### ADR-012-06 · `password_reset_tokens.purpose` — one table, three token flows, a real security boundary (2026-07-09)
- **Decision:** Added a `purpose TEXT NOT NULL DEFAULT 'reset'` column to `password_reset_tokens` (values: `'reset' | 'invite' | 'billing_setup'`) rather than a fourth parallel token table for the Slice 5a.1 billing-setup link. This is the moment the Slice-4a-era ledger predicted: *"upgrade to a purpose column when a consumer queries by it"* — the billing-setup link is that consumer, since it must be able to tell its own tokens apart from reset/invite ones.
- **Why a shared table, not a new one:** reset, invite, and billing-setup tokens are the same primitive (a random bearer token, hashed at rest, tenant-scoped, time-limited) serving different actions. A fourth table would be exactly the "parallel token system" this codebase has already twice declined to build (Slice 4's invite reused reset's machinery for the same reason).
- **The security boundary this column exists to enforce:** a billing-setup token must never be usable to touch a password, and a reset/invite token must never be usable to attach a card or trigger a charge. `confirmPasswordReset` and `peekResetToken` (the `/reset-password` write and read paths) now filter `WHERE purpose != 'billing_setup'` explicitly; `resolveBillingSetupToken` (the `/billing/setup` read path) requires `purpose = 'billing_setup'` explicitly. Verified both directions are actually rejected, not just assumed (`tests/phase12-billing-setup.test.ts`): a billing-setup token handed to `confirmPasswordReset` returns `invalid_token` and leaves the password hash untouched; an invite token handed to `resolveBillingSetupToken` returns `invalid`.
- **Every writer sets its own purpose explicitly** — `requestPasswordReset` writes `'reset'`, `issueInviteToken` writes `'invite'`, `issueBillingSetupToken` writes `'billing_setup'`, all as literal values in the `INSERT` (`src/lib/services/password-reset.ts`), not by relying on the column's default. The default exists solely to backfill rows that predate this migration.
- **Backfill verified, not assumed:** the migration (`016_billing_setup.sql`) runs `ALTER TABLE password_reset_tokens ADD COLUMN purpose TEXT NOT NULL DEFAULT 'reset'`. Reproduced directly — inserted a token row against the pre-016 schema (no `purpose` column at all), then applied 016 unmodified, and confirmed the existing row materializes `purpose = 'reset'` after the fact (SQLite writes the default into every existing row at `ALTER TABLE` time; it is not a lazy read-time default). A pre-existing row that was actually an *invite* gets backfilled as `'reset'` too — a known, harmless imprecision, since `confirmPasswordReset` treats reset and invite identically (the reset/invite distinction was never load-bearing; only the `billing_setup` exclusion is).
- **Context:** Slice 5a.1. Same "log the schema decision, don't just mention it in a slice summary" standard as ADR-012-05 — flagged after the fact once, not left to be re-derived from chat history.

### ADR-012-03 · Stripe billing (⚠ owner-approved spec deviation) + the provisioning DB/Stripe boundary
- **Decision:** Provisioning (`provisionTenant`) creates the tenant + first Admin + locations + requirements template + a **required, validated IANA timezone** in ONE audited DB transaction (through the real services — never dev-seed raw SQL), then attaches a **Stripe customer + SetupIntent** behind an injectable `BillingProvider` seam. Flagged per PATTERN-5; this is a conscious override, not an oversight.
- **The DB-commit / Stripe-call boundary (why it's ordered this way):** a DB transaction can't wrap an external network call. So the core commits FIRST (atomic — any step throwing rolls back the whole tenant, no partial/orphan), THEN Stripe runs. Ordering is tenant-then-customer, so a Stripe customer is only ever created for a tenant that exists → **no orphaned customer-without-tenant**. If Stripe fails after commit, the tenant sits in `provisioning` with `stripe_customer_id` NULL and `billing.attached=false` — **recoverable** (retry is idempotent via `provision:<tenantId>` → reuses the same customer) and **non-billable** (activation, Slice 5, is gated on `stripe_customer_id`), so **no orphaned live tenant**. - **Seam:** `BillingProvider` (createCustomer/createSetupIntent); `StripeBillingProvider` runs only with `STRIPE_SECRET_KEY`, else `NoOpBillingProvider` (dev/test) returns deterministic synthetic ids keyed off the idempotency key — so provisioning works offline and the failure/idempotency paths are testable without the network. Mirrors the Mailer/BlobStore seam.
- **First Admin created WITH a password (loginable now):** there is no internal invite-accept flow (only vendor `/v/[token]`), and password-reset requires an existing hash — so an `invited` admin couldn't set a password. Slice 4's wizard can add an invite option (reusing password-reset as the set-password mechanism). Template rules are attributed to the first Admin (`requirement_rules.created_by → users(id)`; the platform user isn't a tenant user).
- **OPS-7 input closed:** `createTenant` now captures `timezone` (backward-compatible — optional there, REQUIRED + validated in provisioning). The Slice-6 (Phase 11) boundary math already reads it, so a provisioned tenant's expiry math is correct from day one.

## Phase 11 — Launch-prep (infrastructural)

### ADR-011-11 · COI expiry boundary = START of the expiry day, tenant-local (not end-of-day)
- **Decision:** OPS-7 math half. `expiryBoundaryMs(expiry, tz)` resolves a date-only expiry to **00:00 in the tenant's timezone** (start of the expiry day); a full-ISO expiry is honored as-is. Used by the renewal ladder + day-0 flip, Command Center `daysToExpiry`, and reports `daysOut` — one treatment, in `src/lib/time/zone.ts` (same `Intl` primitive as the digest cadence's `localHourInZone`).
- **Context:** Slice 6. The kickoff's framing leaned "valid THROUGH the expiry date → flip at END of that local day." Investigation surfaced two problems with end-of-day.
- **Why start-of-day, not end-of-day:** (1) The **spec wins** (CLAUDE.md) and `Renewal_and_Expiration_Chase` says "Expired **the moment the date passes** … safest compliance posture" and "**past its expiration date**" — i.e. expire *earlier*, don't grant an extra local day of hireability against possibly-lapsed coverage. (2) The current code was already start-of-day (`Date.parse(date-only)` = UTC midnight), so end-of-day would **change UTC behavior** (+1 day) and break `phase7-worker.test.ts:147` (a date-only, UTC-fallback tenant) — tripping the "no-op for UTC / stop if a test changes" rule. The premise that "UTC was already correct, only non-UTC was off" was itself wrong — it was start-of-day for *everyone*; the ONLY bug was the timezone, which is orthogonal to start-vs-end. So: fix the timezone (the real bug), keep the spec's start-of-day semantics. Byte-identical no-op for UTC/null-tz tenants (zero existing-test changes). The user chose this explicitly after the trade-off was surfaced.
- **Not chosen:** end-of-day-local ("valid through") — would be a spec-level semantic change (grant one more day) requiring the spec + those tests updated; flagged, not built.

### ADR-011-01 · ESP is Resend, behind the existing Mailer seam
- **Decision:** Real email goes through Resend, swapped in behind the existing `Mailer` interface (not a new abstraction).
- **Context:** Slice 1. Postmark and Resend were both viable; the whole product value is that chase/invite emails reach the inbox.
- **Why:** Resend's native `Idempotency-Key` header closes the worker double-send window (SEC-2) with zero extra bookkeeping — the worker already threads `notificationId`, which becomes the key directly. Chosen over Postmark for that clean SEC-2 fit. Per-operator DKIM domains deferred (FEAT-13); operator identity rides the From display name on a shared verified domain.

### ADR-011-02 · `SendResult.providerId` is an additive field, not an interface change
- **Decision:** Added an optional `providerId` to `SendResult` rather than treating it as "changing the Mailer interface."
- **Context:** Slice 1. The kickoff said "transport swap, don't change the interface." The cookie-less delivery webhook needs to correlate a bounce to a single tenant-scoped row.
- **Why:** The `send()` method contract and `EmailMessage` are untouched; `providerId` is optional, the NoOp mailer omits it, callers ignore it unless present. It's the only way to persist the ESP message id (`notifications.provider_message_id`) so the webhook can resolve a bounce by **message id, not recipient email** (email collides across tenants). Additive ≠ breaking.

### ADR-011-03 · Login lockout is DERIVED from a rolling window, not a `login_lockouts` table
- **Decision:** No lockout table. `login_attempts` logs failures; the lock is a windowed count. `clearLoginFailuresForEmail` is the unlock seam.
- **Context:** Slice 2 (SEC-9).
- **Why:** Prime Directive — don't invent entities. The lock is fully derivable (count ≥ threshold in window), so a separate row would be speculative. A first-class `login_lockouts` row (with `locked_until`, audited `cleared_by`) only earns its place when a platform dashboard needs "locked until X, unlocked by whom" — a Phase 12 decision, made then. `clearLoginFailuresForEmail(email)` is exactly the seam a future operator-unlock calls. Per-IP scope is a rolling window and is deliberately NOT reset on one success (a valid login must not wipe an attacker's IP count).

### ADR-011-04 · Reset token is hash-only at rest; raw token scrubbed from the notification payload on send AND fail
- **Decision:** `password_reset_tokens` stores only SHA-256(token). The raw token rides `notifications.payload_json` transiently so the worker can render the link, then the worker **nulls it after the send attempt** — on success AND failure. Scoped to `password_reset` only; the invite-token path is deliberately untouched.
- **Context:** Slice 3 + amendment (SEC-8).
- **Why:** The verifier table is hash-only so a DB dump can't reset anyone — but `payload_json` lives in the *same* DB, so a raw token sitting there for the 1h TTL partly undoes that. A reset token **takes over an existing account** on `/confirm` — higher blast radius than an invite token — so it does not inherit the invite pattern's at-rest posture just because delivery plumbing is shared. Scrubbing on failure too (beyond the literal "on success" ask) is required so a failed send doesn't leave the token in a `failed` row for the full TTL. Invite tokens are a separate risk object and a separate decision — not a silent ride-along. **Residual (named):** the raw token is in cleartext between queue and send-attempt (bounded by the worker tick), not for the full TTL.

### ADR-011-05 · Password-reset resolution inherits login's first-match; platform users out of scope
- **Decision:** Reset resolves an email to a tenant user via the **same first-match order login uses**; a multi-tenant email resolves the same way login would. Platform-user reset is not built this phase.
- **Context:** Slice 3.
- **Why:** A reset resolving *differently* than login would be its own bug (SEC-11 carry-forward). Inheriting is correct, not a shortcut. Platform users are engineer-provisioned (operator-assisted model) and the notification worker only delivers to tenant users — platform reset is separate infra, deferred.

### ADR-011-06 · CSRF rejects only `cross-site`; `same-site` is allowed
- **Decision:** Edge middleware rejects `Sec-Fetch-Site: cross-site` (and unknown-Origin on the fallback); `same-site` and `none` pass. CSRF exemptions are **structural** (Bearer / cookie-less), not a hand-maintained path allowlist.
- **Context:** Slice 4 (SEC-12).
- **Why:** The app is single-origin with no untrusted subdomains, so `same-site` (sibling subdomain) is not a live threat, and rejecting it would add risk of false positives for no gain. Making exemptions structural (Bearer isn't ambient; cookie-less can't be CSRF'd) means new cookie-less/token routes are exempt *by construction* — no allowlist to forget to update. **Conditional hardening (tracked in launch-prep):** reject `same-site` too *if* untrusted subdomains are ever introduced. Login is NOT exempt (it's cookie-*setting* → gets the Origin check even without a prior cookie).

### ADR-011-07 · CSRF lives in middleware → matcher must include `/api/*`
- **Decision:** Expanded the middleware matcher to include `/api/*` (it previously excluded it). API routes pass through (never redirected); they own their 401/403 via `getAuth`.
- **Context:** Slice 4. Changed the one test that pinned the old "middleware never runs on /api/" invariant.
- **Why:** "CSRF in edge middleware" and "matcher excludes /api/" are contradictory — the middleware can't protect API mutations it never sees. The matcher change is mandatory for the slice; the test that asserted `/api/` was excluded encoded the *pre-SEC-12* design, so updating it is a design-invariant change, not a test bent to fit a bug.

### ADR-011-08 · Key-version is implicit-v1 for fields (format unchanged), explicit for blobs
- **Decision:** SEC-13 versioning hook. Blob envelope meta stamps `key_version: 1` (free in JSON). Field ciphertext stays 3-part (`iv:tag:ct`) = **v1 by definition**; `decryptField` is version-aware (understands a future `v<N>:iv:tag:ct`) but `encryptField`'s output is unchanged. Migration adds `documents.key_version` (DEFAULT 1). No rotation.
- **Context:** Slice 5 (SEC-13).
- **Why:** Explicitly prefixing `v1:` on fields would change the delimiter arity and break the format-pinning field tests — for zero current benefit (there's no v2 to distinguish from). Unversioned-legacy-as-v1 is the standard pattern; the version tag earns its place only when v2 exists, and the decrypt side already routes it. Decrypt defaults missing/legacy → v1 so all existing data still decrypts. This is a *hook* — `kekForVersion`/`fieldKeyForVersion` throw for unknown versions; rotation itself is deferred.

### ADR-011-09 · Observability scrub is a denylist; its boundary is named, not assumed total
- **Decision:** Sentry `beforeSend` (and self-logged context) run a denylist scrub — redact by sensitive **key name** and by sensitive **digit pattern** (TIN/SSN/EIN/ACH). Alerts pass **IDs only**; the scrub is defense-in-depth on top.
- **Context:** Slice 5 (OPS-12 / SEC-16).
- **Why:** A denylist misses by nature. Named residual: a Sensitive value that is *both* non-digit-shaped (confidential name, lettered account id, base64 secret) *and* under an unrecognized key can slip through. The primary guarantee for the SEC-16/OPS-3 alerts is that callers pass IDs only; the scrub is a safety net for unexpected `captureError` contexts and third-party breadcrumbs, not a total guarantee. Stated so the boundary is known, not oversold.

### ADR-011-10 · Doppler buys at-rest + rotation, not runtime isolation; ALLOWED_ORIGINS is build-time
- **Decision:** Secrets move to Doppler; `env.ts` stays the single chokepoint. Documented honestly in `.env.example`.
- **Context:** Slice 5 (OPS-13).
- **Why:** On a single host, Doppler gives encrypted-at-rest storage + central rotation/audit of secret *values* — it does **not** give runtime process isolation (a compromised app process still reads its own env). Say that, don't oversell it. **Footgun (documented):** Next inlines `process.env` used in *edge middleware* at BUILD time — `ALLOWED_ORIGINS` (and any edge-middleware env) must be present at `next build`, not just injected at runtime, or the edge bundle inlines `undefined` and every Origin-fallback mutation 403s. Build-time vs runtime env are distinct sets in the deploy pipeline.
