## Phase 13 â€” Postgres migration

### ADR-013-01 Â· Data layer migrates SQLiteâ†’Postgres and raw SQLâ†’Kysely, staged module-by-module, before real customer data exists (2026-07-10)
- **Decision:** Migrate the app's data layer off `better-sqlite3` onto Postgres, adopting Kysely (typed query builder) in the same pass rather than staying raw-SQL or introducing a heavier ORM. Target host: Railway (managed Postgres). This is a plan-first, staged migration â€” module by module, never a big-bang rewrite â€” with the SQLite baseline test suite as the correctness contract at every checkpoint (1071 tests as of the Stage 4 commit, not the 1010 figure earlier reports cited â€” see `docs/phase-13-postgres-migration.md`'s baseline-correction note). Each stage is independently committable and must leave the app fully working.
- **Why now, why in one pass with the query-builder switch:** the database is empty (no live customer data), which is the cheapest and safest point in the project's life to absorb a storage-layer change â€” this same reasoning already justified a fresh-baseline-over-migration-porting choice elsewhere in this phase (see the investigation report for the migrations question). Bundling the Kysely adoption into the same pass avoids doing the syncâ†’async rewrite twice (once bare, once again when a query builder is added later) â€” the async ripple is the dominant cost either way, so it should be paid once, deliberately, with type safety as a direct benefit rather than a follow-up nice-to-have.
- **Context:** Phase 13 kickoff, investigation-only pass (no code changed). A full inventory of ~101 files / ~430 direct DB calls across 13 logical modules, a catalog of SQLite-specific constructs (COLLATE NOCASE email lookups, INSERT OR IGNORE/REPLACE, four `rowid`-ordering call sites with no Postgres equivalent, TEXT-as-boolean and TEXT-as-timestamp columns with no CHECK constraints), and a syncâ†’async blast-radius estimate (roughly 250-350 await insertions across ~110-120 files, plus ~13 genuinely structural rewrites at transaction boundaries and N+1-in-map sites) were produced to inform the staged plan. See the investigation report (delivered in chat, not duplicated here) for the full module map, the concrete rework list, and the recommended stage order.
- **Scope boundary:** foundation only â€” when complete, the app must look and behave identically; this is not bundled with any product/design change. `BlobStore` (Backblaze B2 + filesystem adapter) and the background workers' *orchestration* are confirmed independent of the SQL engine choice and are out of scope, though the workers' own DB calls (mostly already-async functions with synchronous `better-sqlite3` calls inside) do get touched as part of whichever module owns their tables.
- **Known hard spots flagged up front, not deferred:** (1) the `billing_snapshots`/`audit_events` `rowid`-tiebreaker pattern needs a real identity/serial column before cutover, in four call sites; (2) `TenantDB.transaction()` and its 9 call sites rely on better-sqlite3's synchronous-callback transaction API (already documented in-repo as the reason the Stripe quantity-sync worker runs as a separate poller) â€” Kysely/Postgres transactions are inherently async, so every transaction boundary is a structural rewrite, not a mechanical `await` add; (3) `COLLATE NOCASE` email/name lookups (8 call sites) need a `lower()`-based rewrite or `citext`; (4) the test suite's true-in-memory-SQLite-per-test isolation (`tests/helpers.ts:setupTestDb`) has no free equivalent on Postgres and needs a deliberate per-test isolation mechanism (transactional rollback or a uniquely-named ephemeral database per run).
- **Confirmed (Stage 0, 2026-07-10): fresh baseline, not a ported one-for-one migration; Kysely, not raw `pg`.** Both were investigation-phase recommendations; Stage 0 built and proved them, so they're adopted, not just proposed. The baseline (`src/migrations-pg/001_baseline.ts` â€” moved here from Stage 0's temporary `db-postgres/` scaffolding once Stage 1 wired it for real) is the cumulative end-state of the 17 SQLite migrations expressed in idiomatic Postgres (booleans as `boolean`, timestamps as `timestamptz`, JSON payloads as `jsonb`, and a real `seq bigserial` tiebreaker column on `billing_snapshots`/`audit_events` replacing `rowid`) â€” verified column-by-column against all 17 files directly, not from a summary. One concrete finding Stage 0 surfaced that refines point (2) above: Kysely's simple callback-style `.transaction()` does NOT auto-nest via savepoints (it throws if called again on an already-open `Transaction`) â€” only the CONTROLLED transaction API (`db.startTransaction()` + `.savepoint(name)`) supports nesting, so `TenantDB.transaction()` in Stage 1 must be built on that API and must itself detect "already inside a transaction" to decide whether to open a new transaction or take a savepoint. Test isolation is solved as a two-layer scheme: one ephemeral database per test run (cloned via `CREATE DATABASE ... TEMPLATE` from a schema-only template, isolating concurrent runs/worktrees from each other) plus a rolled-back transaction per test within that run (isolating tests from each other and reconstructing `:memory:`'s throwaway-per-test guarantee) â€” proven under concurrent execution (two isolation-proof runs launched simultaneously, one from the main tree and one from a separate `git worktree`, against the same Postgres server, each got its own database and neither saw the other's data). Local Postgres 16 (apt-installed on this host) stands in for Railway during Stage 0/1 development; Railway itself is deferred to a later stage per the owner's call, not blocking foundation work.
- **Stage 6a Â· Vendor invite tokens: revoke-on-issue (deliberate security change, NOT behavior-preservation â€” flagged explicitly, not buried in a conversion diff).** Before this stage, a vendor invite token (`invites` table â€” a completely separate mechanism from `password_reset_tokens`, used only by the tokenized vendor portal) was reusable for its full 14-day TTL with no consumption/invalidation step at all: `resendInvite` minted a fresh token without revoking the old one, so two (or more, after repeated resends) tokens could be simultaneously live for the same vendor. - **Cross-stage invariants and landmines, discovered mid-migration (updated through Stage 6b, 2026-07-11) â€” read before converting any further module; full detail and stage status in `docs/phase-13-postgres-migration.md`:**
  1. *Never open a Kysely transaction directly* â€” always `withTransaction()` (`src/lib/db/transaction.ts`), which detects an already-open transaction and takes a `SAVEPOINT` instead of a fresh `BEGIN` (Kysely throws on the latter). Found the hard way: `confirmPasswordReset`'s first version opened one directly and threw inside a test wrapper. `TenantDB.transaction()` delegates to the same helper â€” one implementation, not two to keep in sync.
  2. *`jsonb` columns return already-parsed objects*, not JSON strings â€” never `JSON.parse()` a `*_json` column read. `applyTemplate` (Stage 4) did and threw (`"[object Object]" is not valid JSON`) the moment a real test hit it. Applies to every remaining `*_json` column: `flags_json`, `encryption_json`, `evidence_json`, and the notification/audit `payload_json` columns Stage 8 will touch.
  3. *`bigint`/`bigserial`/`COUNT(*)` results come back as strings* (Postgres precision safety), not numbers â€” cast explicitly wherever used as a number, not just for `ORDER BY`. Regular `integer` columns come back as real numbers; no cast needed there.

  5. *SQLite's `json_extract()`/JSON1 functions have NO Postgres equivalent â€” the Stage 0 investigation missed this entirely, because it only audited DDL for JSON1 usage, not application SELECT queries.* `chase.ts` (Stage 5) tripped over it first; rewritten to Postgres's `->>` jsonb text-extraction operator. **Known-incomplete audit, done post-hoc after Stage 5's code landed, not before:** a repo-wide grep (`json_extract|json_each|json_group_array|json_array|json_object|json_tree|json_quote`) across `src/` turned up two more un-converted call sites, both outside Stage 5's scope:

  6. *`MIN()`/ordering over a jsonb `->>`-extracted value is a lexicographic TEXT comparison, not a chronological one*, unless the extracted value is cast to its real type first (`::timestamptz`, `::date`, `::int`). Silently correct only if every stored value happens to be a fixed-width, zero-padded format â€” not something the query enforces. Found in `chase.ts`'s `chaseExpiryByVendor`/`vendorExpiry` (Stage 5, pre-commit review) via an unpadded date-only fixture (`"2026-9-5"` sorts lexicographically AFTER `"2026-10-01"` as text, despite being chronologically earlier). **The obvious fix is a trap:** an initial version cast to `::timestamptz` and reformatted the result back to text via `to_char(... AT TIME ZONE 'UTC', ...)` â€” this silently turns a DATE-ONLY value into a full timestamp, which breaks `expiryBoundaryMs`'s `DATE_ONLY` regex detection downstream (the tenant-timezone-aware day-boundary math OPS-7 depends on, exercised live in Command Center â€” see the timezone-boundary test in `tests/locations-users-dashboards.pg.test.ts`). Caught before commit, not by a test. **The actual fix never rewrites the value:** `chaseExpiryByVendor` uses `DISTINCT ON (vendor_id) ORDER BY ... (expiration_date)::timestamptz ASC` to pick the chronologically-earliest ROW and return its original, untouched text; `vendorExpiry`'s client-side reduction compares via `Date.parse()` instead of a raw string `<`. `reports/builders.ts` (item 5 above) has the identical `MIN(json_extract(...))` shape and will need the identical `DISTINCT ON` treatment (not `to_char`) when Stage 9 converts it â€” flagged there, not fixed there (out of Stage 5's scope).
  7. *Raw upstream string data needs format normalization at its write choke point, independent of which SQL engine reads it back â€” this is not a Postgres-migration bug, but the migration is what surfaced it.* `expiration_date` values come from Claude Vision's free-form extraction with no format constraint in the tool schema, so a genuinely unpadded/non-ISO value (`"2026-9-5"`) can reach storage. Beyond item 6's lexicographic-`MIN()` risk, an un-normalized value creates a second, subtler seam: `Date.parse()` on a non-conforming string is V8-implementation-defined (resolves in the *process's* local timezone) while Postgres's `::timestamptz` cast resolves in the *Postgres server's* timezone â€” two independently-configured timezones that can silently disagree near a day boundary. Fixed at the single choke point every renewal-chase payload's `expiration_date` flows through â€” `earliestExpiration()` in `src/lib/notifications/renewal.ts` â€” via `toIsoDateStr()` (`src/lib/extraction/extractor.ts`, extended to also zero-pad unpadded ISO-dash dates, not just US slash format, since it previously only handled two shapes and returned `null`, i.e. no-op, for a bare unpadded date). This environment's Node and Postgres both happen to run in UTC, so the divergence isn't locally reproducible today â€” the fix removes the exposure regardless of future deployment timezone config, which is the point.
  8. *`TenantDB.update()` can only express `col = val` equality WHERE predicates* â€” anything needing `IS NULL`, `>`, `<`, or similar drops to the Kysely builder directly on the transaction/db handle, manually re-adding `.where('tenant_id', '=', tenantId)` (the same scoping `TenantDB`'s own helpers enforce internally) rather than reaching for a raw-SQL escape hatch that doesn't exist for writes (`TenantDB.all()`/`.get()` are explicitly READS-only). Found in Stage 6a's `issueInviteToken()` revoke query (`revoked_at IS NULL AND token_expires_at > now()`) â€” likely to recur wherever a future stage needs a conditional bulk UPDATE.
  9. *A jsonb column's WIRE shape (already-parsed object) can silently diverge from what a not-yet-converted consumer still expects (a JSON string), even when the SQL/DB side is entirely correct.* `vendor_locations.flags_json` and `engine_advisories.evidence_json`, read via a converted route (`GET /api/vendors/:id`, Stage 6a) and returned directly in the JSON response, would otherwise double-encode into a real nested object instead of the string `src/app/vendors/[vendorId]/page.tsx` (a NOT-yet-converted frontend page, out of Stage 6a's scope) still does `JSON.parse()` on. Fixed by re-`JSON.stringify()`-ing at the response boundary â€” the route's *internal* handling is fully jsonb-native (never `JSON.parse()`s a read), only the outbound wire contract is deliberately preserved for the sake of an un-converted caller. A reminder that "never `JSON.parse()` a jsonb read" (item 2) doesn't automatically mean "never `JSON.stringify()` on the way back out" â€” those are different questions depending on who's on the other end of the wire. **Temporary, tied to one file â€” see `docs/phase-13-postgres-migration.md`'s Shortcuts & gaps: remove the re-stringify the moment `src/app/vendors/[vendorId]/page.tsx` converts, or it becomes active double-encoding.**
  10. *Inside `withTransaction()`, every nested DB call must take `trx` (or a `TenantDB`/helper built on `trx`) â€” never the outer `db`/`getDb()` handle.* Under SQLite's single stateful connection, a call inside `db.transaction(fn)` that received the outer `db` reference was transactional BY ACCIDENT â€” same connection, same implicit BEGIN/COMMIT, regardless of which JS variable pointed at it. Under Kysely/Postgres, `db` and `trx` are genuinely separate connections: a nested call passing `db` instead of `trx` commits immediately and independently, and survives a rollback of the surrounding transaction â€” a correctness bug, not a style issue. First caught in `confirmPasswordReset` (Stage 3) and again in `createVendorInvite`'s audit-log call (Stage 6a, caught in review before commit, fixed to `logAudit(trx, ...)`). **Audited exhaustively before the Stage 6a commit** â€” every `withTransaction(` call site across Stages 1â€“6a (`locations.ts`'s `bulkImportLocations`, `templates.ts`'s `applyTemplate`, `password-reset.ts`'s `confirmPasswordReset`, `bulk-onboarding.ts`'s `bulkCreateLocationsWithManagers`, `invite-token.ts`'s `issueInviteToken`, `vendors.ts`'s `createVendorInvite`, `provisioning.ts`'s `provisionTenant`) mechanically checked line-by-line for any stray outer-`db` reference inside the callback body, including through nested function calls that themselves take a `db` parameter (`createLocation`, `recordBillingSnapshot`, `createTenant`, `createUser`, `applyTemplate`, `issueInviteToken` â€” each verified to consistently thread whatever handle it's given, never a closure-captured outer one, which is what lets nesting compose correctly at all). Result: clean â€” the `createVendorInvite` fix was the only instance found; no other hits. `provisioning.ts`'s `attachBilling(db, ...)` call is NOT an instance of this pattern (worth naming explicitly, since it's the one place a top-level `db` appears near a transaction) â€” it runs strictly AFTER the `withTransaction` block has already returned/committed (a Stripe API call structurally cannot live inside a DB transaction), not nested inside one passing the wrong handle.

### ADR-012-10 Â· Tenant cockpit rate/fee edits: Stripe-first ordering, immutable-Price swap, subscription-item partial-update reliance (2026-07-09)
- **Decision:** `/platform/tenants/[id]` (Slice 6) gets four controls beyond view: resend the first Admin's credential invite, resend the billing-setup link, edit the per-location rate, edit the one-time setup fee. No new schema â€” `applied_template_id` (Phase 3) was already a column, just never selected by `getTenantById`; no rate-history table was added (the audit trail already covers it, see below).
- **Rate edit ordering â€” Stripe first, local DB second, always:** `updateTenantRate` calls `BillingProvider.updateSubscriptionPrice` BEFORE touching `tenants.monthly_rate_cents`. If Stripe rejects the swap, the local column is not written at all â€” the function returns a structured `{ pushedToStripe: false, error }` (same non-throwing shape as `attachBilling`), never leaves a local rate Stripe isn't actually charging. Verified directly: after an injected Stripe failure, the DB read back is byte-identical to the pre-edit value, and no `tenant.settings_changed` audit event fires.
- **Why a new Price every time, never an edit:** Stripe Prices are immutable (`unit_amount` cannot be changed on an existing Price) â€” same reason `createSubscription` already creates a fresh Price at provisioning. A rate edit is always create-Price + repoint-the-subscription-item, `proration_behavior: 'none'` (identical rule to `updateSubscriptionQuantity`) so the new rate takes effect at the NEXT billing cycle, never a mid-month partial charge.
- **Quantity-sync interaction â€” the highest-risk check in this slice:** the price-swap update (`items: [{id: itemId, price: newPriceId}]`) deliberately omits `quantity`. Stripe subscription-item updates are partial (only passed fields change), so the item's current quantity â€” owned by `quantity-sync.ts`'s poller â€” survives untouched, and the item's own `id` never changes across a price swap (this UPDATEs the existing item, it doesn't delete+recreate it), so `updateSubscriptionQuantity`'s own `items.data[0]?.id` lookup keeps resolving the same item afterward. Reasoning rests on Stripe's documented partial-update semantics and the fact that `updateSubscriptionQuantity` already relies on the identical assumption in reverse (omitting `price` to leave it alone) â€” **not yet re-verified against a live Stripe test-mode call this session** (no key available); carried on `docs/launch-prep.md`'s ledger to fold into the next manual Stripe click-through.
- **Setup-fee guard gates on `stripe_subscription_id IS NOT NULL`, not `lifecycle_state`:** the fee is committed to Stripe as a pending invoice item synchronously inside `attachBilling`, the moment a subscription is created â€” before the tenant necessarily pays (activation waits on the separate `invoice.paid` webhook). A tenant can have a live subscription while still `lifecycle_state='provisioning'`, and the fee is already un-editable at that point; gating on lifecycle state alone would have let an edit through that Stripe would silently ignore. Blocked with an explicit `blockedReason` surfaced to the operator, not a silent no-op.
- **Live subscription status, not a local derivation:** added `BillingProvider.getSubscriptionStatus` and call it on every cockpit page load. Considered "is one more live Stripe call per load too heavy" and concluded no â€” this is a low-traffic, platform-operator-only screen already making one live call (`listRecentInvoices`); an operator screen whose job is showing real payment state must not read a stale local "active" when a customer's card has actually failed.
- **`billing.rate_synced_to_stripe`** logged as its own audit event (distinct from `tenant.settings_changed`, which `updateTenant` already logs with a from/to payload) â€” same reasoning `attachBilling` logs `billing.subscription_created` separately from `billing.customer_attached`: a future audit can tell "did this reach Stripe" apart from "did the DB column change."
- **Resend controls reuse token primitives directly, not `sendUserInvite`:** `resendFirstAdminInvite`/`resendBillingSetupLink` call `issueInviteToken`/`issueBillingSetupToken` directly rather than 5b's `sendUserInvite`, because that function's audit call hardcodes `actorType:'user'` â€” reusing it verbatim would misattribute a platform-initiated resend as a tenant-user action.
- **Context:** Slice 6. Closes the OPS-14 remainder in full (both the invite half from 5b and the billing-setup/rate half here).

### ADR-012-06 Â· `password_reset_tokens.purpose` â€” one table, three token flows, a real security boundary (2026-07-09)

- **Why:** The `send()` method contract and `EmailMessage` are untouched; `providerId` is optional, the NoOp mailer omits it, callers ignore it unless present. It's the only way to persist the ESP message id (`notifications.provider_message_id`) so the webhook can resolve a bounce by **message id, not recipient email** (email collides across tenants). Additive â‰  breaking.

### ADR-011-03 Â· Login lockout is DERIVED from a rolling window, not a `login_lockouts` table
- **Decision:** No lockout table. `login_attempts` logs failures; the lock is a windowed count. `clearLoginFailuresForEmail` is the unlock seam.
- **Context:** Slice 2 (SEC-9).
- **Why:** Prime Directive â€” don't invent entities. The lock is fully derivable (count â‰Ą threshold in window), so a separate row would be speculative. A first-class `login_lockouts` row (with `locked_until`, audited `cleared_by`) only earns its place when a platform dashboard needs "locked until X, unlocked by whom" â€” a Phase 12 decision, made then. `clearLoginFailuresForEmail(email)` is exactly the seam a future operator-unlock calls. Per-IP scope is a rolling window and is deliberately NOT reset on one success (a valid login must not wipe an attacker's IP count).

### ADR-011-04 Â· Reset token is hash-only at rest; raw token scrubbed from the notification payload on send AND fail
- **Decision:** `password_reset_tokens` stores only SHA-256(token). The raw token rides `notifications.payload_json` transiently so the worker can render the link, then the worker **nulls it after the send attempt** â€” on success AND failure. Scoped to `password_reset` only; the invite-token path is deliberately untouched.
- **Context:** Slice 3 + amendment (SEC-8).
- **Why:** The verifier table is hash-only so a DB dump can't reset anyone â€” but `payload_json` lives in the *same* DB, so a raw token sitting there for the 1h TTL partly undoes that. A reset token **takes over an existing account** on `/confirm` â€” higher blast radius than an invite token â€” so it does not inherit the invite pattern's at-rest posture just because delivery plumbing is shared. Scrubbing on failure too (beyond the literal "on success" ask) is required so a failed send doesn't leave the token in a `failed` row for the full TTL. Invite tokens are a separate risk object and a separate decision â€” not a silent ride-along. **Residual (named):** the raw token is in cleartext between queue and send-attempt (bounded by the worker tick), not for the full TTL.

### ADR-011-05 Â· Password-reset resolution inherits login's first-match; platform users out of scope
- **Decision:** Reset resolves an email to a tenant user via the **same first-match order login uses**; a multi-tenant email resolves the same way login would. Platform-user reset is not built this phase.
- **Context:** Slice 3.
- **Why:** A reset resolving *differently* than login would be its own bug (SEC-11 carry-forward). Inheriting is correct, not a shortcut. Platform users are engineer-provisioned (operator-assisted model) and the notification worker only delivers to tenant users â€” platform reset is separate infra, deferred.

### ADR-011-06 Â· CSRF rejects only `cross-site`; `same-site` is allowed
- **Decision:** Edge middleware rejects `Sec-Fetch-Site: cross-site` (and unknown-Origin on the fallback); `same-site` and `none` pass. CSRF exemptions are **structural** (Bearer / cookie-less), not a hand-maintained path allowlist.
- **Context:** Slice 4 (SEC-12).
- **Why:** The app is single-origin with no untrusted subdomains, so `same-site` (sibling subdomain) is not a live threat, and rejecting it would add risk of false positives for no gain. Making exemptions structural (Bearer isn't ambient; cookie-less can't be CSRF'd) means new cookie-less/token routes are exempt *by construction* â€” no allowlist to forget to update. **Conditional hardening (tracked in launch-prep):** reject `same-site` too *if* untrusted subdomains are ever introduced. Login is NOT exempt (it's cookie-*setting* â†’ gets the Origin check even without a prior cookie).

### ADR-011-07 Â· CSRF lives in middleware â†’ matcher must include `/api/*`
- **Decision:** Expanded the middleware matcher to include `/api/*` (it previously excluded it). API routes pass through (never redirected); they own their 401/403 via `getAuth`.
- **Context:** Slice 4. Changed the one test that pinned the old "middleware never runs on /api/" invariant.
- **Why:** "CSRF in edge middleware" and "matcher excludes /api/" are contradictory â€” the middleware can't protect API mutations it never sees. The matcher change is mandatory for the slice; the test that asserted `/api/` was excluded encoded the *pre-SEC-12* design, so updating it is a design-invariant change, not a test bent to fit a bug.

### ADR-011-08 Â· Key-version is implicit-v1 for fields (format unchanged), explicit for blobs
- **Decision:** SEC-13 versioning hook. Blob envelope meta stamps `key_version: 1` (free in JSON). Field ciphertext stays 3-part (`iv:tag:ct`) = **v1 by definition**; `decryptField` is version-aware (understands a future `v<N>:iv:tag:ct`) but `encryptField`'s output is unchanged. Migration adds `documents.key_version` (DEFAULT 1). No rotation.
- **Context:** Slice 5 (SEC-13).
- **Why:** Explicitly prefixing `v1:` on fields would change the delimiter arity and break the format-pinning field tests â€” for zero current benefit (there's no v2 to distinguish from). Unversioned-legacy-as-v1 is the standard pattern; the version tag earns its place only when v2 exists, and the decrypt side already routes it. Decrypt defaults missing/legacy â†’ v1 so all existing data still decrypts. This is a *hook* â€” `kekForVersion`/`fieldKeyForVersion` throw for unknown versions; rotation itself is deferred.

### ADR-011-09 Â· Observability scrub is a denylist; its boundary is named, not assumed total
- **Decision:** Sentry `beforeSend` (and self-logged context) run a denylist scrub â€” redact by sensitive **key name** and by sensitive **digit pattern** (TIN/SSN/EIN/ACH). Alerts pass **IDs only**; the scrub is defense-in-depth on top.
- **Context:** Slice 5 (OPS-12 / SEC-16).
- **Why:** A denylist misses by nature. Named residual: a Sensitive value that is *both* non-digit-shaped (confidential name, lettered account id, base64 secret) *and* under an unrecognized key can slip through. The primary guarantee for the SEC-16/OPS-3 alerts is that callers pass IDs only; the scrub is a safety net for unexpected `captureError` contexts and third-party breadcrumbs, not a total guarantee. Stated so the boundary is known, not oversold.

### ADR-011-10 Â· Doppler buys at-rest + rotation, not runtime isolation; ALLOWED_ORIGINS is build-time
- **Decision:** Secrets move to Doppler; `env.ts` stays the single chokepoint. Documented honestly in `.env.example`.
- **Context:** Slice 5 (OPS-13).
- **Why:** On a single host, Doppler gives encrypted-at-rest storage + central rotation/audit of secret *values* â€” it does **not** give runtime process isolation (a compromised app process still reads its own env). Say that, don't oversell it. **Footgun (documented):** Next inlines `process.env` used in *edge middleware* at BUILD time â€” `ALLOWED_ORIGINS` (and any edge-middleware env) must be present at `next build`, not just injected at runtime, or the edge bundle inlines `undefined` and every Origin-fallback mutation 403s. Build-time vs runtime env are distinct sets in the deploy pipeline.
